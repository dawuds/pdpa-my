## 1. Purpose and Scope

### 1.1 Purpose

This Data Subject Access Request (DSAR) Log serves as the official register for [Organisation Name] to record, track, and manage all rights requests submitted by data subjects under the **Personal Data Protection Act 2010 (PDPA)**. This log ensures that all requests are handled systematically, within statutory deadlines, and with full audit traceability in accordance with the obligations imposed on data processors and data users under Malaysian law.

The log provides the Data Protection Officer (DPO) and relevant stakeholders with a centralised view of the organisation's responsiveness to data subject rights, and constitutes primary evidence of compliance during regulatory audits conducted by the **Personal Data Protection Commissioner (PDPC)**.

### 1.2 Scope

This log applies to:

- All data subject rights requests received by [Organisation Name] through any channel (written, electronic, verbal, or through authorised representatives)
- All personal data held or processed by [Organisation Name] in the course of commercial transactions, as defined under s.4 of the PDPA
- All business units, departments, subsidiaries, and third-party processors acting on behalf of [Organisation Name]
- Requests received from Malaysian residents as well as foreign nationals whose personal data is processed within Malaysia

This log **does not** apply to:

- Requests relating solely to employee personal data processed exclusively for employment purposes (subject to separate HR records procedures)
- Personal data processed for personal, family, or household affairs
- Personal data processed for journalistic, literary, or artistic purposes where a relevant exemption has been claimed

*Each request must be assigned a unique reference number at the point of receipt. This section defines the numbering convention and documents intake details for each request.*

### 4.1 Reference Number Convention

All DSARs are assigned a reference number using the following format:

**`DSAR-[YYYY]-[NNNN]`**

Where:
- `DSAR` — fixed prefix indicating a Data Subject Access Request record
- `[YYYY]` — four-digit calendar year of receipt
- `[NNNN]` — four-digit sequential number, resetting to `0001` at the start of each calendar year

## 5. Request Type Classification

*Accurately classifying the request type determines the applicable statutory provision, processing obligations, and response deadline. Where a single request spans multiple rights, create a separate log entry for each right invoked.*

### 5.1 Request Type Reference

| Request Type | PDPA Basis | Description | Statutory Obligation |
|---|---|---|---|
| **Access Request** | s.30 | Data subject requests to be informed whether personal data is held and to receive a copy | Respond in writing; prescribed fee may apply |
| **Correction Request** | s.34 | Data subject requests correction of inaccurate, incomplete, misleading, or outdated personal data | Correct data or append a statement of contention |
| **Opt-Out Request** | s.43 | Data subject requests cessation of processing that causes or is likely to cause unwarranted damage or distress | Cease or refrain from processing; respond with outcome |
| **Withdrawal of Consent** | s.38 | Data subject withdraws consent previously given for one or more processing activities | Cease processing relying solely on consent; assess alternative lawful basis |
| **Direct Marketing Opt-Out** | s.44 | Data subject requests cessation of all processing for direct marketing purposes | Comply unconditionally; no grounds for refusal |
| **Portability Request** | Best Practice / Contractual | Data subject requests data in a structured, machine-readable format | Not expressly mandated by PDPA; honour where organisational policy commits |
